# Phase 08.1: Content and UI Optimization for Keyword Ranking - Research

**Researched:** 2026-03-03  
**Domain:** SEO landing-page content refresh + conversion-focused UI redesign  
**Confidence:** HIGH

## Summary

Phase 08.1 should execute as a focused optimization sprint across high-intent landing templates (`home`, `features`, `industries`, `compare`, `contact`) using the current contract-first SEO architecture already in this repo. The fastest path is not introducing new CMS infrastructure; it is upgrading existing typed content contracts and template layouts while preserving guardrails (`seo:check`, `seo:check-links`, `seo:check-messaging`).

For ranking impact, the research signal is consistent: prioritize people-first usefulness, make title/heading intent explicit, and preserve crawlable internal linking with descriptive anchor text. For conversion impact, keep proof-led copy and clearer CTA hierarchy on mobile and desktop while retaining instrumentation and deterministic link paths.

**Primary recommendation:** Build one optimization foundation plan (priority map + copy contract guardrail), then execute two parallel redesign plans: hub/home surfaces and detail/conversion templates.

## Architecture Patterns

### Pattern 1: Contract-first copy updates
**What:** Keep page intent/copy primitives in typed modules and consume in template routes/components.  
**When to use:** Multi-page refreshes where consistency matters.

### Pattern 2: Template-level redesign with invariant guards
**What:** Redesign page sections and CTA hierarchy while running SEO/messaging/link scripts as release gates.  
**When to use:** Visual/copy improvements that should not break canonical or internal-link contracts.

### Pattern 3: Priority-map-driven execution
**What:** Generate a deterministic optimization queue from weekly SEO report artifacts, then map updates by template/page type.  
**When to use:** Choosing what to rewrite first instead of broad, unfocused edits.

### Anti-patterns to avoid
- Page-local copy edits that bypass shared messaging contracts.
- UI redesign that changes internal-link structure without link checks.
- Keyword-stuffed headings or generic anchors that reduce relevance clarity.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| SEO policy validation | Ad-hoc per-page checks | Existing SEO contract scripts | Already enforces canonical/indexable invariants |
| Internal-link QA | Manual link spot-checking only | `seo:check-links` guardrail | Deterministic crawl-path validation at scale |
| Copy consistency | Untracked static strings everywhere | `lib/messaging.ts` + guardrail checks | Prevents template drift and promise-order regressions |

## Common Pitfalls

### Pitfall 1: Rewrites that weaken search clarity
**What goes wrong:** New hero/body copy sounds polished but loses concrete query intent.  
**How to avoid:** Keep primary intent terms explicit in title/H1/lead body and validate against template keyword targets.

### Pitfall 2: Link architecture regressions during redesign
**What goes wrong:** Visual refresh removes or weakens key internal links.  
**How to avoid:** Preserve feature/industry/compare cross-links and run link guardrail scripts on every pass.

### Pitfall 3: Copy quality drift across templates
**What goes wrong:** Different pages diverge from ICP phrasing and promise sequence.  
**How to avoid:** Expand messaging contract checks to include newly refreshed templates and CTA variants.
